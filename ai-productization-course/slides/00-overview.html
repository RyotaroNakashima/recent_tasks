学習ゴール

  • LLM/RAG/MLOpsを学ぶ理由を市場価値と実務責任の観点で説明できる。
  • OCR/画像AI経験を、LLMアプリ設計・評価・運用へ接続して語れる。
  • 本教材で扱う範囲と、次回以降に回すAgent/MCP/ガバナンス等の境界を理解する。

なぜ今LLM/RAG/MLOpsを学ぶのか

企業導入で求められるのは「モデルを試す人」ではなく、業務データ、権限、評価、運用まで含めてAIを業務に載せる人です。

実務判断

LLMの性能だけで採用判断しない。データ鮮度、根拠提示、監査ログ、費用上限を満たせるかで判断する。

失敗例

デモでは便利だが、更新されない社内文書・権限漏れ・回答の根拠不明で本番停止になる。

OCR/画像AIエンジニアから伸ばす自然なルート

OCRでは入力品質、前処理、モデル、後処理、評価を分けて改善します。RAGも同じく、文書整形、chunking、retrieval、generation、評価、運用を分解します。

モデルを作る人からAIを業務に載せる人へ

  • モデル単体: 精度指標、推論速度、学習データ。
  • LLMアプリ: ユーザー権限、文書鮮度、プロンプト、検索品質、コスト、ログ、fallback。
  • 面接では「精度改善」だけでなく「本番で壊れない設計」を語る。

今回扱う範囲

  • LLMアプリとRAGの基本構造
  • retrieval設計: chunking、embedding、vector DB、metadata、reranking
  • generation設計: prompt、citation、grounding、structured output
  • 評価設計: golden dataset、retrieval評価、answer評価、LLM-as-a-judge
  • LLMOps: API、Docker、ログ、監視、コスト、rollback

今回あえて深掘りしない範囲

  • Agent詳細: tool planningやmulti-agentは次段階。
  • MCP: ツール接続の標準化として紹介に留める。
  • AIガバナンス: リスク管理の入口のみ。
  • クラウド詳細: AWS/GCP/Azure固有構成は扱わない。
  • データエンジニアリング詳細: ETL基盤は概念レベル。

学習後の到達イメージ

「RAGを作りました」ではなく、なぜそのchunkingか、なぜそのmetadataか、どう評価し、障害時に何を切り分けるかを説明できる状態を目指します。

設計説明 = 要件 + データ特性 + 検索戦略 + 生成制御 + 評価 + 運用

推奨学習順

  1. 00で地図を掴む。
  2. 01でRAG全体構造を理解する。
  3. 02-03でretrievalとgenerationを分けて設計する。
  4. 04で評価可能な形に落とす。
  5. 05で本番運用の最低ラインを確認する。
  6. 06-07でケースとアウトプットへ変換する。

面接・職務経歴書での語り方

「OCRの誤差分析を、RAGのretrieval/generation評価分離に応用できます。入力ノイズ、前処理、後段の業務影響まで見て改善できます」と語る。

年収800万円以上を狙うなら、技術名の羅列ではなく、品質保証・運用・事業影響まで設計できることを示す。
