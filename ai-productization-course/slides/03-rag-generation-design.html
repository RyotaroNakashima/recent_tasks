03-rag-generation-design

Generation設計：根拠に忠実な回答を作る

実務で説明・設計・評価・運用できる力へ接続する章です。

操作: ← / → キー、または下部ボタンで移動。

学習ゴール

  • prompt constructionとcontext packingを説明できる。
  • grounding、faithfulness、hallucinationを区別できる。
  • citation、回答不能判定、structured outputを設計できる。

なぜ学ぶのか

本番RAGでは、部品の名前を知るだけでは不十分です。ユーザー要件、データ特性、失敗時の影響から、どの設計を選ぶかを判断します。

prompt construction

system promptで役割・禁止事項・回答形式を固定し、user promptで質問を渡し、contextには検索結果とmetadataを構造化して入れます。

system prompt / user prompt

System: 根拠に含まれない内容は推測しない。回答にはcitationを付ける。
User: {question}
Context: [{doc_id, title, updated_at, text}]

context packing

top-kをそのまま詰め込むと重要箇所が埋もれます。重複除去、権限確認、最新文書優先、長すぎるchunkの要約を検討します。

citation設計

回答文のどの主張がどの文書に基づくかを示します。OCR文書ではページ、領域、OCR信頼度も表示すると確認しやすくなります。

回答不能判定と「知らない」と言わせる設計

根拠が不足する時は回答しないことを仕様にします。検索スコア閾値、根拠数、矛盾検出を使い、fallbackとして問い合わせ先や確認依頼を出します。

grounding / faithfulness / hallucination

  • grounding: 回答が提供contextに基づく度合い。
  • faithfulness: 根拠に忠実で矛盾しない度合い。
  • hallucination: 根拠なしの断定、数字の捏造、存在しない規程の引用。

structured output / JSON schema

後段システムへ渡す場合はJSON schemaで型を固定します。自由文より検証・再試行・監査が容易です。

function callingとの接続

RAGだけで答えず、在庫照会、CRM更新、メール下書きなどは関数呼び出しへ分離します。副作用がある操作は人間確認を挟みます。

ユーザー権限に応じた回答制御

retrievalで権限filterし、generationでも「表示可能な根拠のみで答える」制約を入れます。権限外情報の存在を示唆しないUIも重要です。

長文/短文回答と日本語RAG

日本語では敬語、曖昧な主語、表記揺れが問題になります。回答長、箇条書き、要約レベルを明示し、引用箇所と推論を分離します。

OCRノイズが生成に与える影響

「0/O」「1/l」などの誤認識は会社名、金額、日付の誤回答に直結します。低信頼OCRは断定せず、確認を促すプロンプトにします。

OCR/画像AI経験との接続

OCRの前処理、信頼度、誤差分析、項目一致率の考え方は、RAGの文書整形、metadata、failure taxonomy、評価設計へ転用できます。

面接・職務経歴書での語り方

「モデル精度だけでなく、データ品質・検索品質・生成の忠実性・運用監視まで分解して改善できます」と、担当範囲をプロダクト品質に広げて表現します。

ミニ演習

社内文書RAG用promptを作り、根拠なし回答禁止、citation必須、回答不能時の文言、JSON出力形式を追加して改善する。

まとめ

  • 技術要素は必ず業務要件と失敗リスクに紐づけて選ぶ。
  • RAG品質は検索、生成、評価、運用の掛け算で決まる。
  • OCR経験者はノイズと評価を扱える点が強みになる。

確認問題

  1. この章の主要技術を3つ挙げ、それぞれの実務判断を説明してください。
  2. 典型的な失敗例を2つ挙げ、検知方法と改善策を述べてください。
  3. OCR/画像AI経験と接続できる点を1つ説明してください。
