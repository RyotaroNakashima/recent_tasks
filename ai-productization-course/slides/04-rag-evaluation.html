04-rag-evaluation

RAG評価設計：本番導入を判断する

実務で説明・設計・評価・運用できる力へ接続する章です。

操作: ← / → キー、または下部ボタンで移動。

学習ゴール

  • retrieval評価とanswer評価を分離できる。
  • golden datasetとfailure taxonomyを設計できる。
  • LLM-as-a-judgeの使いどころと危険性を説明できる。

なぜ学ぶのか

本番RAGでは、部品の名前を知るだけでは不十分です。ユーザー要件、データ特性、失敗時の影響から、どの設計を選ぶかを判断します。

LLM/RAG評価が難しい理由

回答は一意に決まらず、検索結果・prompt・model version・文書更新で変動します。だから評価データ、指標、ログを固定して比較します。

従来ML/OCR評価との違い

OCRは文字認識精度や項目一致率を測れます。RAGは検索が正しいか、回答が正しいか、根拠に忠実か、UXとして役立つかを分けます。

評価の分解

retrieval評価
generation評価
end-to-end評価
UX評価
business KPI

retrieval指標

  • recall@k: 正解文書が上位k件に入るか。
  • precision@k: 上位k件の関連度。
  • MRR: 最初の正解順位。
  • nDCG: 関連度の高い文書が上位にあるか。
  • context precision / context recall: LLMへ渡した文脈の質。

answer指標

  • correctness: 事実として正しい。
  • groundedness: contextに基づく。
  • faithfulness: 根拠と矛盾しない。
  • relevance: 質問に答えている。
  • completeness: 必要事項が揃う。
  • conciseness: 過不足なく短い。

golden datasetの作り方

{question, user_role, expected_doc_ids, must_include_facts,
 forbidden_claims, expected_answer, difficulty, category}

簡単・難しい・権限境界・古い文書・OCRノイズを含めます。

failure taxonomy

検索漏れ、ノイズ混入、古い文書、権限違反、根拠なし生成、矛盾、過剰回答、形式違反に分類すると改善担当が明確になります。

LLM-as-a-judge

大量評価の補助に有効ですが、judge prompt、評価基準、モデル変更に影響されます。合否の最終判断は人間評価やサンプル監査と組み合わせます。

judge promptとバイアス

あなたは評価者です。回答がcontextに含まれる根拠だけで
質問に答えているかを1-5で採点し、根拠文を引用せず理由を述べる。

長い回答を高評価しがち、権威ある文体に騙される等のバイアスに注意します。

regression test / red teaming / 本番ログ改善

promptやindex更新のたびにgolden datasetで回帰評価し、悪意ある質問や権限境界をred teamingします。本番ログは匿名化し、失敗例をgolden datasetへ昇格します。

OCR/画像AI経験との接続

OCRの前処理、信頼度、誤差分析、項目一致率の考え方は、RAGの文書整形、metadata、failure taxonomy、評価設計へ転用できます。

面接・職務経歴書での語り方

「モデル精度だけでなく、データ品質・検索品質・生成の忠実性・運用監視まで分解して改善できます」と、担当範囲をプロダクト品質に広げて表現します。

ミニ演習

名刺OCR + 社内CRM検索RAGについて、代表質問、正解文書、retrieval指標、answer指標、失敗分類、改善サイクルを設計する。

まとめ

  • 技術要素は必ず業務要件と失敗リスクに紐づけて選ぶ。
  • RAG品質は検索、生成、評価、運用の掛け算で決まる。
  • OCR経験者はノイズと評価を扱える点が強みになる。

確認問題

  1. この章の主要技術を3つ挙げ、それぞれの実務判断を説明してください。
  2. 典型的な失敗例を2つ挙げ、検知方法と改善策を述べてください。
  3. OCR/画像AI経験と接続できる点を1つ説明してください。
