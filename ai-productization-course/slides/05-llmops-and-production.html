05-llmops-and-production

LLMOps/MLOps：PoCを本番へ載せる

実務で説明・設計・評価・運用できる力へ接続する章です。

操作: ← / → キー、または下部ボタンで移動。

学習ゴール

  • PoCと本番運用の違いを説明できる。
  • API化、Docker化、ログ、監視、コスト、fallbackを設計できる。
  • 障害時にretrieval/prompt/model/data/権限を切り分けられる。

なぜ学ぶのか

本番RAGでは、部品の名前を知るだけでは不十分です。ユーザー要件、データ特性、失敗時の影響から、どの設計を選ぶかを判断します。

PoCと本番運用の違い

PoCは動けばよい。本番は止まらず、漏らさず、説明でき、戻せる必要があります。評価・監視・rollbackがないRAGは本番準備不足です。

API化とDocker化

FastAPI等で入力・認証・trace_id・エラー形式を固定し、Dockerで実行環境を再現します。Docker化は入口であり、監視やsecret管理が別途必要です。

環境変数とsecret管理

API key、DB URL、model名、index名、prompt versionを環境変数化します。secretはコード・ログ・ブラウザへ出さず、ローテーション可能にします。

logging / tracing / monitoring

{trace_id, user_id_hash, query, retrieved_doc_ids, scores,
 prompt_version, model_version, latency_ms, token_cost, answer_status}

latency管理とtoken cost管理

検索、rerank、生成のどこが遅いかを分けて測ります。context量、model選定、cache、batch処理で費用と遅延を制御します。

rate limit / retry / timeout

外部LLM APIは失敗します。指数バックオフ、最大再試行、タイムアウト、ユーザー向けエラー文を決めます。無限retryは障害を増幅します。

fallback / cache / batch処理

LLM失敗時は検索結果だけ表示、定型回答、問い合わせ誘導にfallbackします。FAQはcache、夜間index更新はbatchで扱います。

index更新とrollback

新indexを別名で構築し、評価後にaliasを切り替えるblue/green的運用にします。異常時は旧indexへ戻せるようにします。

model/version管理とprompt/version管理

回答品質が変わった時、model、prompt、retriever、index、データのどれが原因か追えるようversionをログへ残します。

evaluation pipelineとhuman feedback loop

CIでgolden datasetを回し、本番の低評価・問い合わせ・修正依頼をレビューして評価データに戻します。人間確認は品質改善の燃料です。

障害時の切り分け

  • retrievalが悪い: 正解文書がtop-kにない。
  • promptが悪い: 根拠はあるが回答形式や断定が悪い。
  • modelが悪い: 同じpromptでモデル差が出る。
  • dataが古い: 文書更新がindexへ反映されない。
  • 権限設計が悪い: access_group filter漏れ。

OCR/画像AI経験との接続

OCRの前処理、信頼度、誤差分析、項目一致率の考え方は、RAGの文書整形、metadata、failure taxonomy、評価設計へ転用できます。

面接・職務経歴書での語り方

「モデル精度だけでなく、データ品質・検索品質・生成の忠実性・運用監視まで分解して改善できます」と、担当範囲をプロダクト品質に広げて表現します。

ミニ演習

RAGアプリの本番運用チェックリストを、API、Docker、secret、ログ、監視、cost、latency、fallback、rollback、評価の観点で作る。

まとめ

  • 技術要素は必ず業務要件と失敗リスクに紐づけて選ぶ。
  • RAG品質は検索、生成、評価、運用の掛け算で決まる。
  • OCR経験者はノイズと評価を扱える点が強みになる。
