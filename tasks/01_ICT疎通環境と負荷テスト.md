# ICT疎通環境と負荷テスト

## 概要
2026-04-02のICT向け疎通確認環境提供、続く負荷テスト準備、そして性能説明までが1つの塊になっているテーマ。実作業、段取り、説明準備、依頼待ちが混在している。

## 現状の事実
- 2026-04-02に、ICT向け疎通確認環境の提供が必要。
- 疎通確認環境は、大石さんのリポジトリをそのまま使う想定。
- 負荷テストではリポジトリ相当が2系統必要だが、そのうち1系統は疎通確認環境の流用で賄える見込み。
- もう1系統は、大石さんのリポジトリをコピーし、必要な変数や名前を変えて立ち上げる案が頭の中にある。
- 負荷テスト当日は、午前中に自分担当テストを終わらせたい認識がある。
- APIは2系統立てる想定で、それぞれに対してCPU / GPU計測を回す想定。
- GPU計測コマンドは、既存のGPU計測用 `ipynb` に詳細がある認識。
- CPU / メモリ計測コマンドも、既存の `shell` や `calculate CPU` 系の `ipynb` から引けそうな認識。
- 処理時間は、デバッガーやsync log系から取得できる見込み。
- 第3OCRの処理時間が、ICT基準の `1時間4000枚` `1台あたり9.9秒以内` を超えそうという試算がある。
- 速度低下の理由として、全文OCR時にカテゴリ名を出すようにした変更があり、出力量増加が性能に影響している認識がある。
- その変更で精度がどれだけ上がったかの情報は、諸村さんからまだもらっていない。
- 2026-01末から2026-02末に、さらに遅いバージョンで運用しても大きな問題の声は上がらなかった認識がある。
- 2026-07に速度向上検証の結果を入れ込む想定がある。

## いまの仮方針
- 2026-04-02の環境提供は、大石さんのリポジトリを使って先に通す。
- 負荷テスト用の2系統目は、既存環境のコピーで作り、変数名や識別名だけを調整する。
- 計測コマンドは新規作成せず、既存のNotebook / スクリプトから参照元を特定して使う。
- ICTには、現行試算では基準超過の可能性があることを先に伝えたうえで、理由と暫定運用案をセットで説明する。
- 暫定運用案は、2026-04から2026-07までは現速度での運用を相談し、2026-07の速度改善で巻き返す方向。
- 必要なら、下村さんや清水さんに説明補助や意思決定支援の場を設ける。

## 引っかかり・止まりポイント
- CPU / GPU計測コマンドの具体的な参照元がまだ頭の中で曖昧。
- 実作業だけでなく、ICTへの説明準備と「なぜ事前に分からなかったのか」への備えが重くなっている。
- 速度低下の理由は頭にあるが、根拠になる精度向上幅が未回収。
- 暫定運用の相談は筋があるが、どこまで相手に受け入れてもらえるか読めず重い。
- 負荷テスト準備、負荷テスト当日進行、結果説明が1つの塊になっていて、頭の中で未分解。
