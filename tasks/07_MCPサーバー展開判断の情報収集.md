# MCPサーバー展開判断のための情報収集

## 概要
コハマさんが作っているMCPサーバー相当のものを、上層部へ展開するかどうかを考える前段として、他部署がGemini Enterpriseまわりで似た仕組みを作っていないか確認するテーマ。明日の終礼での聞き方が未確定なことが、主な引っかかり。

## 現状の事実
- コハマさんは、以前AIエージェント検討チームのリーダーだった。
- コハマさんがMCPサーバーのようなものを作っている。
- それを会社の上の方へ展開するかどうかで迷っている認識がある。
- 他部署が、Gemini Enterpriseまわりで社内DBをMCP化するような近い取り組みをしている可能性があるという話が出ている。
- もし他部署が近いものを作っているなら、こちら側の展開を見直す可能性がある。
- 明日の終礼ミーティングのような場で、他部署の取り組み有無を確認したい。
- ただし、こちら側の事情を出しすぎると、`じゃあそっちで全部やってください` `こっちは引くのか` のような流れになるのを怖がっている。
- そもそもこのテーマは、社長発の `DBをMCPサーバー化した方がいいのでは` という話が起点だったが、なぜ作ったかの説明自体が自分の中で少し曖昧になっている。

## いまの仮方針
- 明日の場では、意思決定ではなく情報収集に徹する。
- 聞く内容は `Gemini Enterpriseまわりで社内DBや既存データ活用をどこまで検討・開発しているか` に絞る。
- こちらが似たものを作っていることや、展開可否で迷っていることは、その場では出さない。
- 情報を得た後に、`展開継続 / 保留 / 要整理` の仮置きをする。

## 引っかかり・止まりポイント
- 手札を出しすぎて、判断を急かされるのが怖い。
- 単なる興味本位に見える聞き方も避けたい。
- 逆に、こちらが引く口実を探しているように見えるのも避けたい。
- 元の起点説明が曖昧で、自分の中でも筋道が少しぼやけている。

## 未決事項 / 誰に確認するか
- 明日の相手
  - 他部署が実際に何をどこまでやっているか。
- コハマさん / 自分側で後で詰めること
  - その情報を踏まえ、展開継続・保留・要整理のどこに置くか。
- 要確認
  - 終礼ミーティング名の正式名称。原文では `タブショット` など音声文字起こしらしい揺れがある。

## 次アクション
- 明日の場で使う質問文を1つ決める。
- `なんで聞いているのか` と返された時の返答を用意する。
- 終礼後に、得た情報をもとに `展開継続 / 保留 / 要整理` の仮置きをする。

  - 2026-04-01の終礼想定。
- 関係者
  - コハマさん
  - 他部署側の担当者
  - 上層部

  - 聞き方が固まらないと、終礼で情報が取れずに再度持ち越しになりやすい。
