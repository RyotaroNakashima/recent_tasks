# LLM引き継ぎ: 教授定例PowerPoint資料作成指示

## 1. 依頼内容

以下の内容に沿って、教授定例用のPowerPoint資料を作成してください。

目的は、報告者が教授に対して、AIエージェントによる学習自動化ワークフローの進捗、データ準備エージェントの最終検証結果、今後の方針、相談点を分かりやすく説明することです。

成果物は `.pptx` 形式のPowerPointファイルにしてください。

## 2. 資料の前提

### 報告者の立場

- 報告者は研究・開発の担当者
- 教授定例で、現在の進捗と相談点を報告する
- 聞き手は教授
- 目的は、細かい実装ログの共有ではなく、現状の判断材料と今後の方針について意見をもらうこと

### 重要な方針

- 読み手が一歩ずつ理解できるように、論理の飛躍を避ける
- 「何を作っているか」「なぜその評価設計にしたか」「結果はどうだったか」「なぜそう解釈するか」「次に何をするか」を順番に示す
- 1回目の検証結果は詳しく話さない
- 2回目の最終検証結果を中心に話す
- チーム方針として、最終的な検証結果をメインで報告する
- 1回目の検証を長く説明すると主旨がずれるため、必要最小限に留める

## 3. 研究・開発テーマ

現在、AIエージェントによって機械学習モデルの学習サイクルを自動化するワークフローを構築している。

対象としている主なタスクは、名刺画像から会社名・部署名・役職・氏名・メールアドレスなどのカテゴリを検出する物体検出モデルである。モデルはRT-DETRを想定している。

全体としては、学習結果の分析、評価結果の分析、次サイクルの戦略立案、追加データの選定、再学習をAIエージェント群で回す構成になっている。

## 4. エージェント構成

以下のエージェント群で学習改善サイクルを回す。

| エージェント | 役割 |
| --- | --- |
| 学習分析エージェント | 学習時のMLflowメトリクス、学習条件、パラメータ、学習曲線、精度値などをもとに、どのような学習だったかを分析する |
| 評価分析エージェント | 評価データの各画像について、VLMを用いて誤検出・未検出・誤分類などの理由を分析し、類似する失敗傾向を集約・要約する |
| 戦略立案エージェント | 学習分析と評価分析の結果を統合し、次の学習サイクルでどのような方針を取るべきかを提案する |
| データ準備エージェント | 戦略立案エージェントの方針に基づき、次サイクルで追加すべき学習データをデータプールから選定する |
| モデル改善エージェント | モデル改善方針を扱う想定。ただし、今回のデータ準備エージェント検証では使用しない |
| 再学習エージェント | 選定された追加データを用いて再学習を実行する |
| スクリプト改善エージェント | 学習・評価などのスクリプト改善を担当する想定 |

## 5. 学習自動化ループ

想定しているループは以下。

1. 学習を実行する
2. 学習分析エージェントが学習結果を分析する
3. 評価分析エージェントが評価画像ごとの失敗理由を分析し、傾向を集約する
4. 戦略立案エージェントが次サイクルの学習方針を立てる
5. データ準備エージェントが追加すべきデータを選定する
6. 毎サイクル固定枚数のデータを学習データに追加する
7. 再学習エージェントが再学習を実行する
8. 同じサイクルを複数回繰り返す

現時点では、1サイクルごとに100枚のデータを追加する想定。

## 6. 直近の開発対象

直近では、データ準備エージェントを構築している。

データ準備エージェントの目的は、戦略立案エージェントが出した方針に基づいて、データプールの中から次に追加すべき学習データを選ぶことである。

追加対象のデータは、あらかじめ用意したデータプールの中から選択する。データプール内の各画像には、事前にメタデータを付与しておく。

## 7. データ準備エージェントで使う情報

データ準備エージェントには以下の情報を渡す。

- 現在の学習データに含まれるメタデータ分布
- 評価で誤ったデータに含まれるメタデータ分布
- 評価分析エージェントがまとめた失敗傾向
- 戦略立案エージェントが提案した次サイクルの学習方針

これらをもとに、データ準備エージェントが「どのようなメタデータ特徴を持つデータを何枚追加するか」を決定する。

メタデータ例:

- 名刺画像の縦横比
- 縦長・横長などのレイアウト
- 文字が縦書きか横書きか
- 部署と役職の位置関係・距離
- 画像がぼやけているか
- 画像がかすれているか
- 名刺画像に含まれる特徴

メタデータは、ルールベースで付与するものと、VLMに画像を読ませて特徴抽出するものがある。

## 8. 前回定例で共有済みの内容

前回は、主に以下を報告済み。

- エージェント全体構成をおおむね整理した
- データ準備エージェントの設計方針を説明した
- データ抽出ロジックをおおむね固めた
- 実装は6割程度完了しており、今後も実装を進める予定と報告した

今回時点では、データ準備エージェントの実装は完了し、精度検証に移れる段階まで進んだ。

そのため、今回の定例では「データ準備エージェントの精度検証結果」を中心に報告する。

## 9. 検証対象タスク

検証対象は、名刺画像に対する物体検出モデル。

対象モデル:

- RT-DETR

検出対象:

- 会社名
- 部署名
- 役職
- 氏名
- メールアドレス
- 電話番号
- FAX番号
- 住所関連項目
- その他、名刺上のカテゴリ

名刺画像上の各項目を矩形で囲み、カテゴリを検出・認識するタスクとして扱っている。

## 10. 評価設計

データ準備エージェントは、学習サイクルごとにデータプールから追加データを選定し、学習データへ追加する。

ただし、データを追加すると精度が上がる可能性があるため、精度向上が以下のどちらによるものかを切り分ける必要がある。

- データ準備エージェントが有効なデータを選べているため精度が上がった
- 単に学習データ数が増えたため精度が上がった

そのため、比較対象として以下の2条件を設定した。

| 条件 | 内容 |
| --- | --- |
| VLM選択 | 学習分析・評価分析・戦略立案の結果を踏まえ、VLMを用いて追加データ100枚を選定する |
| ランダム選択 | データプールからランダムに100枚を選定する |

各サイクルで100枚ずつデータを追加し、VLM選択とランダム選択の精度推移を比較する。

## 11. 1回目検証の扱い

1回目の検証は資料内で詳しく説明しない。

必要な場合のみ、以下のように短く触れる。

- 初期検証では、ベース学習量が不足しているとランダム追加でも精度が上がりやすい可能性が見えた
- そのため、最終検証ではベース学習量を増やし、ランダム追加だけでは上がりにくい条件を作った

1回目の具体条件や結果を長く説明しないこと。

1回目の条件:

- ベース学習データ数: 6,000枚
- エポック数: 15エポック
- 各サイクルの追加枚数: 100枚

1回目の結果:

- ランダム選択の方が精度が高いように見えた
- VLM選択の方は、ランダム選択より精度が低かった

ただし、PowerPointではこの内容を独立スライドにしない方がよい。

## 12. 2回目の最終検証

今回の中心は2回目の最終検証。

### 条件変更の意図

1回目の結果を踏まえ、ランダムに100枚追加するだけでは精度が上がりにくい状態を作る必要があると考えた。

つまり、ベースモデルを十分に学習させたうえで、VLM選択による「弱点を狙ったデータ追加」の効果が見えやすい条件にすることを狙った。

### 条件

- ベース学習データ数: 12,000枚
- エポック数: 30エポック
- 1回目と比べて、データ数を2倍、エポック数を2倍にした
- 学習量としては約4倍に増やした
- 各サイクルの追加枚数: 100枚

### 結果

- ランダム選択では精度が上がらず、横ばいになった
- VLM選択でも精度が上がらず、横ばいになった

### 解釈

- ランダムに100枚追加するだけでは精度が上がらない状態を作る、という点では想定通りだった
- 一方で、VLM選択でも精度が上がらなかった
- そのため、現時点のデータ準備エージェントは、物体検出タスクにおいて十分に有効な追加データを選べていない可能性がある

## 13. 現在確認していること

現在、VLMの出力が本当に質の高いものになっているかを確認している。

確認したい観点:

- VLMが評価結果の誤り理由を適切に理解できているか
- 戦略立案エージェントが妥当な改善方針を出せているか
- データ準備エージェントが戦略に対応したデータを選べているか
- 選定されたデータが、実際にモデルの弱点を補うものになっているか

VLM出力データはまとまっているため、今後はその内容を見ながら、どこで質が落ちているかを分析する。

## 14. 現時点の仮説

現時点では、物体検出タスク自体が複雑すぎるため、AIエージェントが誤り傾向や改善方針を十分に理解できていない可能性がある。

理由:

- 名刺カテゴリが18カテゴリ程度あり、分類ルールが複雑
- 電話番号とFAX番号など、見た目や文脈が近いカテゴリの切り分けが難しい
- 部署名と役職名など、意味的に近いカテゴリの切り分けが難しい
- 住所について、どこまでを都道府県・市区町村・番地などとして扱うかが分かりづらい
- GTアノテーションルール自体が複雑
- 現時点では、GTアノテーションルールを十分にエージェントへ渡せていない

本来であれば、以下のような情報をエージェントに十分与える必要がある。

- GTアノテーションルール
- 各カテゴリの定義
- 紛らわしいカテゴリの判断基準
- どのような場合にどのラベルを付けるべきか
- 正解アノテーションの具体例

これらが不足しているため、AIエージェントが物体検出モデルの失敗傾向を正しく理解し、適切な学習戦略に落とし込むのは難しい可能性がある。

## 15. 今後の方針

物体検出タスクでのデータ準備エージェントの効果検証は、一旦ここまでとする。

次に、より単純な分類タスクで、AIエージェントによる学習自動化ワークフロー全体の効果を検証する方針。

次の検証対象:

- 名刺画像の表裏判定モデル

タスク内容:

- 表
- 裏
- 裏かつ白紙

の3クラス分類。

モデル:

- MobileNet系の単純なクラス分類モデル

このタスクは、物体検出よりも分類ルールが単純であり、AIエージェントが誤り傾向や改善方針を理解しやすいと考えられる。

現在進めている作業:

- 表裏判定モデルをAIエージェントによる学習自動化フローに適合させる
- 表裏判定用のメタデータプールを準備する
- 処理構成の整理
- リファクタリング
- データ準備エージェントを表裏判定タスクで再実行する準備

## 16. 教授への相談点

相談点は以下の3つ。

### 相談1: 検証対象を物体検出から分類タスクへ移す方針について

物体検出タスクでは、カテゴリ数やアノテーションルールが複雑で、AIエージェントが失敗傾向を十分に理解できていない可能性がある。

そのため、まずは表裏判定のような単純な分類タスクで、AIエージェントによる学習自動化フローが機能するかを検証したい。

この方針について妥当か意見をもらいたい。

### 相談2: 物体検出タスクの検証を継続する場合に必要な追加情報について

物体検出で継続する場合、GTアノテーションルールやカテゴリ定義をエージェントに詳細に渡す必要があると考えている。

どの程度までルールを明文化してエージェントに与えるべきか相談したい。

### 相談3: データ準備エージェントの評価設計について

VLM選択とランダム選択を比較する評価設計自体は妥当と考えているが、他に比較すべきベースラインや評価指標があるか相談したい。

PowerPointは10枚程度にする。

### 1枚目: 表紙

タイトル:

- データ準備エージェントの精度検証と今後の方針

サブタイトル:

- AIエージェントによる学習自動化ワークフロー

補足:

- 教授定例資料
- 日付
- 今回の主題: 2回目の検証結果を中心に方針を相談

### 2枚目: 本日の結論

伝えること:

- 物体検出タスクでは、VLM選択による明確な精度向上は確認できなかった
- タスクとGTルールが複雑で、エージェントが誤り傾向を十分に理解できていない可能性がある
- 次は、より単純な表裏判定の3クラス分類で、学習自動化ワークフローの効果を再検証する

### 3枚目: 全体像

タイトル例:

- 全体像: 学習改善サイクルをエージェントで回す

内容:

- 学習
- 学習分析
- 評価分析
- 戦略立案
- データ準備

メッセージ:

- 目的は、単にデータを増やすのではなく、分析結果に基づいて「次に効くデータ」を選ぶこと

### 4枚目: 評価設計

タイトル例:

- 評価設計: データ選択の効果とデータ追加の効果を切り分ける

内容:

- 精度が上がった場合、それが「選び方の質」なのか「枚数が増えた効果」なのかを分ける必要がある
- 比較条件A: VLM選択
- 比較条件B: ランダム選択
- 各条件で100枚追加し、精度推移を比較

### 5枚目: 最終検証条件

タイトル例:

- 最終検証条件: ベース学習を十分に行った状態で比較

内容:

- 狙い: ランダムに100枚追加するだけでは精度が上がりにくい状態を作り、VLM選択の効果を見えやすくする
- ベース学習データ: 12,000枚
- エポック数: 30 epoch
- 追加枚数: 100枚 / cycle

補足:

- 初期検証の詳細は割愛し、最終検証としてこの条件を中心に報告する

### 6枚目: 最終検証結果

タイトル例:

- 最終検証結果: 精度推移と比較表

このスライドは、報告者があとから精度グラフと精度表を貼り付ける前提にする。

必須要素:

- 左側に大きな「精度グラフ貼り付け枠」
- 右側に「精度表貼り付け枠」
- 上部に短い要点:
  - 最終検証では、VLM選択・ランダム選択ともに精度は横ばい
- 下部に読み取り:
  - ランダム追加だけでは上がりにくい条件は作れたが、VLM選択も改善に結びつかなかった

注意:

- このページには、あとからグラフと表を貼るための余白を大きく取る
- 長い説明文を置かない

### 7枚目: 結果の解釈

タイトル例:

- 結果の解釈: 問題は「選定ロジック」か「タスク理解」か

内容:

| 区分 | 内容 | 読み取り |
| --- | --- | --- |
| 観測 | ランダム追加では横ばい | 単純な追加枚数の効果は抑えられている |
| 観測 | VLM選択でも横ばい | 弱点を補うデータを選べていない可能性 |
| 仮説 | 物体検出タスクが複雑 | 誤り傾向・GTルールをエージェントが理解しきれていない可能性 |

下部メッセージ:

- 次に見るべき点は、VLM出力と選定データが本当にモデルの弱点に対応していたか

### 8枚目: 物体検出タスクで難しい点

タイトル例:

- 物体検出タスクで難しい点

内容:

- カテゴリ数: 名刺内の検出対象が約18カテゴリあり、判断対象が多い
- 境界が曖昧: 部署名・役職名、電話番号・FAX番号など、近いカテゴリの切り分けが難しい
- GTルール: 住所や項目範囲など、正解アノテーションのルール自体が複雑

下部メッセージ:

- GTルールやカテゴリ定義を、エージェントに十分渡せていない
- 誤り理由の理解からデータ選定までのどこかで、品質が落ちている可能性がある

### 9枚目: 今後の方針

タイトル例:

- 今後の方針: まずは単純な分類タスクで効果を検証

内容:

- 物体検出での追加改善を続ける前に、ワークフロー自体が機能する条件を確認する
- 次の対象: 表裏判定モデル
- タスク: 表 / 裏 / 裏かつ白紙の3クラス分類
- モデル: MobileNet系分類モデル

補足:

- 分類ルールが単純で、エージェントが誤り傾向を理解しやすい
- データ準備エージェントの有効性を切り分けて確認しやすい
- 現在、表裏判定用メタデータプールと処理構成を準備中

### 10枚目: 相談したいこと

タイトル:

- 相談したいこと

内容:

1. 検証方針
   - 物体検出での検証を一旦区切り、表裏判定の分類タスクで効果検証する方針は妥当か
2. 情報設計

3. 評価設計
   - VLM選択とランダム選択以外に、追加すべきベースラインや評価指標があるか

## 18. デザイン指定

### 全体

- 教授向けの研究報告資料として、落ち着いた見た目にする
- 白基調
- 黒または濃いグレーの本文
- 強調色はオレンジ系を1色だけ使う
- 装飾は少なめ
- 余白を十分に取る
- 1スライド1メッセージを意識する

### 避けること

- 文字を詰め込みすぎない
- 1回目検証を長く説明しない
- 原因を断定しすぎない
- 「VLMが悪い」と単純化しない
- 論理を飛ばしていきなり方針転換しない
- 派手なグラデーションや装飾を使わない

### 使うとよい表現

- 「可能性がある」
- 「現時点の仮説」
